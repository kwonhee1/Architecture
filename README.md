# Shopping — 아키텍처 학습 프로젝트

하나의 **고정된 쇼핑몰 스펙**(기능 명세 · 기능 흐름도 · API 명세)을 정해 두고,
**동일한 요구사항을 여러 아키텍처로 각각 구현**하며 아키텍처를 비교·학습하는 것이 목표입니다.

> 스펙은 모든 구현이 공유하고, 구현체는 아키텍처별 폴더로 분리합니다.
> "무엇을 만드는가"는 고정하고 "어떻게 구조화하는가"만 바꿔 가며 학습합니다.

---

## 📌 목표

- 같은 도메인(회원 / 쿠폰 / 상품·옵션 / 주문)을 서로 다른 구조로 구현
- 각 아키텍처의 **책임 분리 방식, 의존성 방향, 트랜잭션 경계**를 직접 비교
- "정답 아키텍처"를 찾는 것이 아니라, 각 스타일의 **트레이드오프를 체감**하는 것

구현을 시작하기 전에 이 문서들을 먼저 읽어 주세요. **스펙은 구현마다 바뀌지 않습니다.**

### 도메인 요약

- **User** — 회원가입 / 로그인(`UserId` 쿠키) / 정보 조회 / 포인트 충전
  - 회원가입 시 첫 쿠폰 `{ "회원 가입 쿠폰", 1000 }` 자동 발급
- **Coupon** — 금액 할인 쿠폰(% 아님), 사용 가능 쿠폰 조회 (생성은 회원가입 시 내부 처리)
- **Product / Option** — 상품 등록·수정, 옵션 생성·수정·삭제, 상품 리스트·상세·내 상품 조회
  - 상품 등록 시 기본 옵션 `{ "기본 옵션", 0, 재고 0 }` 함께 생성
- **Order** — 주문(쿠폰 사용 · 포인트 차감 · 재고 차감), 구매자/판매자 주문 조회, 주문 취소(롤백)

### 인증

- 로그인 성공 시 `UserId=%d` 형태의 **쿠키**를 발급
- 인증이 필요한 API는 이 쿠키로 사용자를 식별하고 권한을 확인
- (명세 요구사항) password는 인코딩하지 않고 저장/비교하며, 어떤 응답에도 반환하지 않음

## 🔑 이 프로젝트의 핵심 — 복잡한 로직을 어떻게 풀어내는가

단순 CRUD가 아니라, **여러 도메인이 얽힌 복잡한 로직을 각 아키텍처가 어떻게 구조적으로 풀어내는지**가 이 프로젝트의 핵심입니다.
대표적으로 **주문 / 주문 취소** 흐름은 재고 · 포인트 · 쿠폰 · 주문이 한 트랜잭션 안에서 함께 움직입니다.

![주요 기능 flowchart](./주요%20기능%20flowchart.png)

- **주문 (order-01)** — 로그인 확인 → 재고 확인 → 포인트 확인 → 쿠폰 확인 → (재고 차감 · 포인트 차감 · 쿠폰 사용) → 주문 가격 계산 → 주문 생성
  - 어느 한 단계라도 실패(재고 부족 / 포인트 부족 / 잘못된 쿠폰)하면 **전체를 취소**하고 실패 사유를 반환
- **주문 취소 (order-04)** — 로그인/소유자 확인 → 쿠폰 사용 취소 · 포인트 롤백 · 재고 롤백 → 주문 삭제

이 흐름에서 **여러 도메인(order · product · coupon · user)의 로직을 어디에 두고, 트랜잭션 경계와 도메인 간 의존을 어떻게 정리하는지**가
아키텍처마다 달라집니다. 같은 flowchart를 각자의 방식으로 구현한 결과를 비교하는 것이 학습의 중심입니다.

> 세부 규칙(동일 옵션 합산, 쿠폰 적용 후 금액 > 0, order 존재 시 수정 제약 등)은 [`기능명세서`](./기능명세서) 참고.

## 🏛️ 아키텍처별 구현

각 폴더가 동일 스펙에 대한 **하나의 아키텍처 구현**입니다.

| 폴더 | 아키텍처 | 핵심 아이디어 |
|------|----------|---------------|
| [`Layerd/`](./Layerd) | Layered (계층형) | Controller → Service → Repository 3계층 |
| [`Facade/`](./Facade) | Application / Domain Service | 서비스를 application service(실행 순서·트랜잭션·외부 호출)와 domain service(도메인 로직)로 분리 |
| [`DomainPort/`](./DomainPort) | Ports & Adapters (Hexagonal) | 도메인이 port 인터페이스에만 의존하고 adapter가 구현 |
| [`DDD/`](./DDD) | Domain-Driven Design | Aggregate / Entity / VO 중심 모델링 |

각 아키텍처의 상세 규칙과 실행 방법은 **해당 폴더의 README**를 참고하세요.

## ✅ 테스트

모든 구현은 **동일한 시나리오** [`test.http`](./test.http)를 통과해야 합니다.
(어떤 아키텍처로 구현하든 겉으로 드러나는 동작은 같아야 하므로, 테스트도 공통입니다.)

### 진행 방법

1. 검증할 아키텍처 구현의 서버를 실행 (실행법은 해당 폴더 README 참고, 포트 `8080`)
2. CLI가 `test.http`의 요청을 **위에서부터 순서대로** 호출
3. 각 요청의 payload · response 와 `# 기대:` 주석(성공 판정 기준)을 비교
4. 재실행 시에는 이전 실행의 데이터와 겹치지 않도록 **DB를 초기화**

- 각 시나리오는 **self-contained** — 필요한 user · 상품 · 옵션 · 포인트 · 주문을 시나리오 안에서 직접 생성합니다.
- 로그인 시 발급되는 `UserId` 쿠키를 저장해 이후 요청에 사용하고, 액터(구매자/판매자/제3자)가 바뀌면 재로그인합니다.
- db/env를 직접 읽지 않고 **API 응답만으로** 검증합니다.

### 주요 테스트 (각 구현이 반드시 통과)

| 구분 | 검증 내용 |
|------|-----------|
| 회원/인증 | 회원가입 시 가입 쿠폰 자동 발급, email 중복(409), 로그인 실패(401), 미인증 접근(401), password 미반환 |
| 경계값 | 판매 금액 > 0, 재고 >= 0, (추가금액 + 상품금액) > 0, 충전 포인트 > 0 |
| 권한 | 상품/옵션은 생성자만 수정·삭제(403), 주문 취소는 본인만(403) |
| **주문 (핵심)** | 쿠폰 유/무 각각의 정상 주문, **쿠폰 없이 포인트 부족 / 쿠폰 있어도 포인트 부족**, 재고 부족, 쿠폰 적용 후 금액 ≤ 0, 동일 옵션 개수 합산 |
| **실패 원자성** | 주문 실패 시 포인트 · 재고 · 쿠폰이 **변하지 않음** |
| order 존재 제약 | 주문이 있는 상품의 금액 수정 불가(설명은 가능), 옵션 수정/삭제 불가(재고 수정은 가능) |
| **주문 취소 롤백** | 취소 시 포인트 · 재고 복원, 쿠폰 사용 가능 상태로 복원, 남은 포인트 반환 |

> 전체 케이스(S01 ~ S53)는 [`test.http`](./test.http)에 있습니다.
